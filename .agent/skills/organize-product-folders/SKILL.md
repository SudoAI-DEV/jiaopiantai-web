# 客户产品文件夹整理

## 概述

当用户把新的客户图片、备注或群文件夹交给你时，使用这个 skill。

目标不是直接生成电商图，而是先把客户资产整理成统一、可继续流转的结构，供后续 `generate-clothing-ecommerce-images`、`styled-clothing-shoot` 等流程直接使用。

## 目标结构

**就地整理**——产品留在客户给出的批次目录中，不迁移到别处：

```text
客户/{客户名}/{年份季节_批次}/
└── {场景目录}/
    └── {产品编码}/
        ├── 微信图片_xxx.jpg
        ├── 微信图片_xxx.jpg
        ├── 配置.yaml
        └── 生成/
            └── batch_NN/
                ├── scenes.json
                └── scene_01~10.png
```

例如：
```text
客户/花姐&寅寅/2026_春夏_第二批/
├── 都市街拍/
│   ├── 2-SJ-01/
│   ├── 2-SJ-02/
│   └── ...
├── 艺术建筑/
│   ├── 2-YJ-01/
│   └── ...
└── 田园自然/
    ├── 2-TR-04/
    └── ...
```

## 硬性规则

1. **就地整理，不迁移**
   - 产品目录必须留在用户指定的批次文件夹内
   - 将客户原始子文件夹（如"新建文件夹"、带要求描述的文件夹）重命名为产品编码
   - **不要**把文件移动到 `产品 2/` 或其他位置

2. **产品编码规范**
   - 格式：`{批次}-{场景缩写}-{序号}`
   - 批次编号：从 2 开始（产品 2）
   - 场景缩写：
     - SJ = 都市街拍（含原室外街拍）
     - HA = 海边艺术
     - TR = 田园自然
     - YJ = 艺术建筑
   - 旧缩写（仅限已有产品）：DS = 都市松弛
   - 序号：01, 02, 03... 顺序递增
   - 同一场景目录如果跨批次已有编号（如田园自然在第一批用到 2-TR-03），新批次接续编号（从 2-TR-04 开始）
   - 例如：`2-HA-02`, `2-TR-04`, `2-SJ-05`, `2-YJ-01`

3. **不要再使用 `原图/` 子文件夹**
   产品挂拍图、细节图、背面图都直接放在 `{产品编码}/` 根目录下。

4. **优先复用最新文件夹内容**
   如果客户给了新的群图、更新版本、补拍细节或新备注，始终以最新文件夹为准。

5. **每个产品目录必须有 `配置.yaml`**（唯一配置文件，合并原 `要求.yaml` 的职责）
    解析文件夹名称并结合图片分析，写入以下信息：
    - `product_code` - 产品编码
    - `client` - 客户名
    - `season` - 年份季节（含批次）
    - `scene` - 当前场景名
    - `folder_requirements` - 具体要求

6. **所有图片记录与图片视觉分析，一并写入 `配置.yaml`**（由后续流程使用）
    - **全部图片都要被加入**：不再挑选 2 张图，必须将文件夹中所有的产品图全数列入 `selected_images`。
    - 同时写入 `model_image` 与 `custom_requirements`
    - **必须对每张图片进行视觉分析**：用大模型查看所有图片，将所有核心视觉信息转为文本全数列至 `image_descriptions`，并汇总大纲至 `clothing_summary`（见第 4 步）
    - 目的：下游 skill 直接消费涵盖了所有图的文本描述，不再重复读取图片，大幅节省 token

7. **客户备注不能丢**
   - 若客户目录内已有 `摘要.md`、`要求.md`、`要求.markdown.md` 等文件，要保留

## 整理流程

### 第 1 步：识别客户资产

检查：
- 客户名
- 批次目录路径
- 场景目录名（子文件夹层）
- 每个产品的原始文件夹名（包含客户要求）
- 图片文件
- 备注文件
- 是否已有 `配置.yaml`（若有旧的独立 `要求.yaml`，计划合并进来）

### 第 2 步：确定编码起点

- 检查同一场景目录在其他批次中已使用的最大编号
- 新批次的产品接续编号，不与已有编码冲突
- 有具体要求的文件夹排前面，"新建文件夹"排后面

### 第 3 步：就地重命名目录

- 将原始文件夹名（如"裤套装 上衣不用扎进去"、"新建文件夹 (3)"）重命名为产品编码（如 `2-DZ-01`）
- 把产品图片直接放在产品目录根部
- 删除"原图"这一层结构假设

### 第 4 步：生成 `配置.yaml`（含客户要求与视觉分析）

**重要：文件夹名包含所有客户要求**

客户给的所有要求都在产品文件夹名字中体现，格式如：
`客户/{客户名}/{年份季节}/{场景目录}/{具体要求}/`

例如：`客户/花姐&寅寅/2026_春夏_第一批/海边艺术/前领口钉珠 后领口系带/`

**注意**：如果文件夹名是"新建文件夹"等无意义名称，`folder_requirements` 字段应为空。

#### 4a. 图片视觉分析（核心优化步骤）

**目的**：在整理阶段一次性用大模型看完所有产品图，将视觉信息转为文本，下游 skill 直接消费文本，不再重新读图。

对每一张图片，用大模型查看并提取：

1. **图片角色**：正面整体 / 背面整体 / 正面细节 / 背面细节 / 下装平铺 / 挂拍全貌 等
2. **画面内容详细描述**（≥50 字），必须覆盖：
   - 服装品类、颜色、面料质感
   - 廓形、版型、长度
   - 领口、肩线、袖型
   - 腰部结构（收腰/松腰/腰带）
   - 下摆、裙摆/裤腿形态
   - 所有装饰元素的**名称、精确位置（正面/背面/两侧）、形态**
   - 口袋、拉链、纽扣、开衩等功能性结构
   - 图案/印花的具体描述（颜色、风格、分布）

然后综合所有图片，生成结构化的 `clothing_summary`。

#### 4b. 生成 `配置.yaml`

`配置.yaml` 是每个产品的**唯一配置文件**，合并了客户要求、选图配置和视觉分析。

```yaml
# ── 客户要求（原 要求.yaml 的内容） ──
product_code: 2-SJ-01
client: 花姐&寅寅
season: 2026_春夏_第二批
scene: 都市街拍
folder_requirements: 半裙套装 上衣扎进去拍

# ── 选图与模特配置 ──
# 注意：应当将该产品目录下所有图片都列入 selected_images
selected_images:
  - 微信图片_xxx.jpg
  - 微信图片_yyy.jpg
  - 微信图片_zzz.jpg
selected_image_notes:
  - 正面整体主图
  - 背面整体
  - 细节补充图
model_image: 客户/花姐&寅寅/models/模特2号.png
custom_requirements:
  - 半裙套装
  - 上衣扎进去拍

# ── 图片视觉分析（大模型看图后生成，下游 skill 直接消费文本） ──

image_descriptions:
  - file: 微信图片_xxx.jpg
    role: 正面整体
    description: |
      挂拍正面照。深藏蓝色亚麻半裙套装。
      上衣：圆领短袖，宽松版型，前中无扣，下摆自然收边。
      半裙：高腰A字，及膝长度，左侧隐藏拉链，两侧斜插口袋。
      面料：亚麻混纺，有自然褶皱纹理，不透。
      整体颜色：深藏蓝色，无图案。
  - file: 微信图片_yyy.jpg
    role: 背面整体
    description: |
      挂拍背面照。
      上衣背面：与正面同版型，无特殊设计。
      半裙背面：后中无开衩，后腰无装饰，无口袋。

clothing_summary:
  category: 半裙套装（上衣+半裙）
  color: 深藏蓝色
  fabric: 亚麻混纺，有自然褶皱纹理
  silhouette: 上衣宽松，半裙A字收腰
  length: 上衣及腰，半裙及膝
  key_features:
    - 圆领短袖上衣
    - 高腰A字半裙
    - 左侧隐藏拉链
    - 两侧斜插口袋
    - 有内衬，不透
  front_back_differences: |
    正面：圆领，无扣，自然下摆，斜插口袋
    背面：无特殊设计，无开衩，无口袋
  decoration_elements: []
    # 若有装饰，格式如下：
    # - name: 珍珠链
    #   position: 仅正面
    #   location: 领口V字沿边
    #   form: 连续链状，约15颗

notes:
  - 若后续补拍了更清晰图片，应更新 selected_images 并重新生成 image_descriptions 和 clothing_summary
```

若当前产品没有额外客户定制要求，`custom_requirements` 可为空数组，但 `selected_images`、`model_image`、`image_descriptions` 和 `clothing_summary` 仍必须填写。

## 后续流程衔接

后续生成类 skill 必须：
- 读取 `配置.yaml` 获取客户要求、模型、图片配置和**视觉分析文本**
- **如果 `配置.yaml` 包含 `image_descriptions` 和 `clothing_summary`，直接消费文本，不再重新读取图片**
- 仅在以下情况才需重新看图：`配置.yaml` 缺少视觉分析字段、客户补拍了新图但未更新分析、用户明确要求重新分析
- 优先使用 `model_image` 指定的模特图
- 以 `selected_images` 作为主要产品图输入
- 把 `folder_requirements` 写入场景设计和 prompt
- 统一使用 `scene` 字段承载当前场景名
- 不再从 `原图/` 子目录取图

### `配置.yaml` 的职责归属

- `配置.yaml` 的**首要生成 / 维护责任**在本 skill
- 当客户图片刚整理入库、补拍细节、替换模特、调整主图选择时，应优先更新这里的 `配置.yaml`
- 后续 `generate-clothing-ecommerce-images` 应在继续执行前检查并按需补全 / 更新 `配置.yaml`
- `styled-clothing-shoot` 主要消费 `配置.yaml`；它不能默默跳过该文件，但也不应把“长期维护配置”当成自己的主职责

## 处理原则

- 就地整理，产品留在批次目录内，不迁移到其他位置
- 目录整理要稳定、可迁移、可脚本化
- 尽量不改动客户原始文件名（图片文件名），避免后续人工对照困难
- 若旧备注与最新图片矛盾，以最新图片和最新备注为准
- 若 `配置.yaml` 已存在，优先增量更新，不随意重写用户已确认内容
- 若遇到旧项目仍有独立 `要求.yaml`，将其内容合并进 `配置.yaml` 后可删除
