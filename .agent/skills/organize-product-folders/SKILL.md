# 客户产品文件夹整理

## 概述

当用户把新的客户图片、备注或群文件夹交给你时，使用这个 skill。

目标不是直接生成电商图，而是先把客户资产整理成统一、可继续流转的结构，供后续 `generate-clothing-ecommerce-images`、`styled-clothing-shoot` 等流程直接使用。

## 目标结构

**就地整理**——产品留在客户给出的批次目录中，不迁移到别处：

```text
客户/{客户名}/{年份季节_批次}/
└── {场景目录}/
    └── {产品编码}/
        ├── 微信图片_xxx.jpg
        ├── 微信图片_xxx.jpg
        ├── 要求.yaml
        ├── 配置.yaml
        └── 生成/
            └── batch_NN/
                ├── scenes.json
                └── scene_01~10.png
```

例如：
```text
客户/花姐&寅寅/2026_春夏_第二批/
├── 大自然/
│   ├── 2-DZ-01/
│   ├── 2-DZ-02/
│   └── ...
├── 艺术建筑/
│   ├── 2-YJ-01/
│   └── ...
└── 田园自然/
    ├── 2-TR-04/
    └── ...
```

## 硬性规则

1. **就地整理，不迁移**
   - 产品目录必须留在用户指定的批次文件夹内
   - 将客户原始子文件夹（如"新建文件夹"、带要求描述的文件夹）重命名为产品编码
   - **不要**把文件移动到 `产品 2/` 或其他位置

2. **产品编码规范**
   - 格式：`{批次}-{场景缩写}-{序号}`
   - 批次编号：从 2 开始（产品 2）
   - 场景缩写：
     - DS = 都市松弛
     - HA = 海边艺术
     - TR = 田园自然
     - DZ = 大自然
     - YJ = 艺术建筑
     - SJ = 室外街拍、咖啡馆
   - 序号：01, 02, 03... 顺序递增
   - 同一场景目录如果跨批次已有编号（如田园自然在第一批用到 2-TR-03），新批次接续编号（从 2-TR-04 开始）
   - 例如：`2-HA-02`, `2-TR-04`, `2-DZ-05`, `2-YJ-01`

3. **不要再使用 `原图/` 子文件夹**
   产品挂拍图、细节图、背面图都直接放在 `{产品编码}/` 根目录下。

4. **优先复用最新文件夹内容**
   如果客户给了新的群图、更新版本、补拍细节或新备注，始终以最新文件夹为准。

5. **每个产品目录必须同时有 `要求.yaml` 和 `配置.yaml`**
    解析文件夹名称，提取以下信息写入：
    - `product_code` - 产品编码
    - `client` - 客户名
    - `season` - 年份季节（含批次）
    - `scene` - 当前场景名
    - `folder_requirements` - 具体要求

6. **`selected_images` 固定选 2 张，写入 `配置.yaml`**（由后续流程使用）
    - 一张优先锁定完整正面 / 主视角
    - 一张优先锁定背面、关键细节、下装结构或客户重点补拍位置
    - 同时写入 `model_image` 与 `custom_requirements`，供后续生成类 skill 直接消费

7. **客户备注不能丢**
   - 若客户目录内已有 `摘要.md`、`要求.md`、`要求.markdown.md` 等文件，要保留

## 整理流程

### 第 1 步：识别客户资产

检查：
- 客户名
- 批次目录路径
- 场景目录名（子文件夹层）
- 每个产品的原始文件夹名（包含客户要求）
- 图片文件
- 备注文件
- 是否已有 `要求.yaml`
- 是否已有 `配置.yaml`

### 第 2 步：确定编码起点

- 检查同一场景目录在其他批次中已使用的最大编号
- 新批次的产品接续编号，不与已有编码冲突
- 有具体要求的文件夹排前面，"新建文件夹"排后面

### 第 3 步：就地重命名目录

- 将原始文件夹名（如"裤套装 上衣不用扎进去"、"新建文件夹 (3)"）重命名为产品编码（如 `2-DZ-01`）
- 把产品图片直接放在产品目录根部
- 删除"原图"这一层结构假设

### 第 4 步：生成 `要求.yaml` 与 `配置.yaml`

**重要：文件夹名包含所有客户要求**

客户给的所有要求都在产品文件夹名字中体现，格式如：
`客户/{客户名}/{年份季节}/{场景目录}/{具体要求}/`

例如：`客户/花姐&寅寅/2026_春夏_第一批/海边艺术/前领口钉珠 后领口系带/`

**注意**：如果文件夹名是"新建文件夹"、"新建文件夹 (1)"、"新建文件夹 (2)"等无意义名称，`folder_requirements` 字段应为空，不要把这些无意义名称放入。

生成 `要求.yaml`：

```yaml
product_code: 2-DZ-01
client: 花姐&寅寅
season: 2026_春夏_第二批
scene: 大自然
folder_requirements: 半裙套装 上衣扎进去拍
```

或无具体要求时：

```yaml
product_code: 2-DZ-07
client: 花姐&寅寅
season: 2026_春夏_第二批
scene: 大自然
folder_requirements:
```

再生成 `配置.yaml`：

```yaml
product_code: 2-DZ-01
scene: 大自然
selected_images:
  - 微信图片_xxx.jpg
  - 微信图片_yyy.jpg
selected_image_notes:
  - 正面整体主图
  - 背面或关键细节补充图
model_image: 客户/花姐&寅寅/models/模特2号.png
custom_requirements:
  - 半裙套装
  - 上衣扎进去拍
notes:
  - 若后续补拍了更清晰的背面或细节图，应优先更新 selected_images
```

若当前产品没有额外客户定制要求，`custom_requirements` 可为空数组，但 `selected_images` 与 `model_image` 仍应明确写入。

## 后续流程衔接

后续生成类 skill 必须：
- 统一使用 `scene` 字段承载当前场景名

- 先读 `要求.yaml` 获取客户要求
- 读取 `配置.yaml` 获取模型和图片配置
- 优先使用 `model_image` 指定的模特图
- 以 `selected_images` 作为主要产品图输入
- 把 `folder_requirements` 写入场景设计和 prompt
- 不再从 `原图/` 子目录取图

### `配置.yaml` 的职责归属

- `配置.yaml` 的**首要生成 / 维护责任**在本 skill
- 当客户图片刚整理入库、补拍细节、替换模特、调整主图选择时，应优先更新这里的 `配置.yaml`
- 后续 `generate-clothing-ecommerce-images` 应在继续执行前检查并按需补全 / 更新 `配置.yaml`
- `styled-clothing-shoot` 主要消费 `配置.yaml`；它不能默默跳过该文件，但也不应把“长期维护配置”当成自己的主职责

## 处理原则

- 就地整理，产品留在批次目录内，不迁移到其他位置
- 目录整理要稳定、可迁移、可脚本化
- 尽量不改动客户原始文件名（图片文件名），避免后续人工对照困难
- 若旧备注与最新图片矛盾，以最新图片和最新备注为准
- 若 `要求.yaml` 或 `配置.yaml` 已存在，优先增量更新，不随意重写用户已确认内容
