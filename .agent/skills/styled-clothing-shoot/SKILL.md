# 服装风格拍摄

## 概述

当用户希望生成 AI 服装电商图时，使用这个 skill。Agent 直接完成场景设计（不依赖单独的场景生成脚本），然后调用渲染脚本输出图片。

**核心原则**：这些图片的目标是**卖衣服**，不是展示风景。所有构图、光线、道具、姿态决策都必须服务于服装销售。

**硬性禁令**：`scenes.json` 绝对不能通过批量脚本、临时辅助脚本、固定模板、批量改字、成组复制 prompt 的方式生成。每一款衣服都必须先实际查看该款服装图，再结合 `要求.yaml`、`配置.yaml`、模特图和风格参考图，直接由 Agent 单独写出属于这款衣服自己的 20 个场景。哪怕同一风格下连续多款产品，也不能直接复用上一款的场景文案骨架。

**配置职责说明**：`配置.yaml` 不应在流程里被默默忽略。它的首要维护责任在前置流程（优先是 `organize-product-folders`，其次是作为第一生成层的 `generate-clothing-ecommerce-images`）；本 skill 默认消费它。如果当前产品目录缺失 `配置.yaml`，不能直接绕过，必须先补齐或更新后再继续场景设计。

## 工作流程

### 第 1 步：接收输入

用户通常会提供：
- **产品目录**：优先按当前真实结构 `客户/{客户名}/{season}/{风格名}/{产品编码}/` 理解，例如 `客户/花姐&寅寅/2026_春夏_第二批/田园自然/2-TR-06/`。若遇到旧项目，也可能是历史结构 `客户/{客户名}/产品 2/{风格名}/{产品编码}/`；无论哪种结构，都以 `要求.yaml` 所在目录作为当前产品根目录。产品图直接放根目录，不再使用 `原图/` 子目录，通常包含正面、背面、细节
- **客户要求**：读取同目录下的 `要求.yaml`，获取 client、season、style、folder_requirements
- **模特参考图**：位于 `客户/{客户名}/models/`，优先以 `配置.yaml` 中指定的 `model_image` 为准
- **风格选择**：A / B / C / D / E / F / G
- **风格参考图**：位于 `客户/{客户名}/风格参考/风格X_XXX/`
- **产品配置**：优先读取同目录下的 `配置.yaml`；若缺失，先补齐配置，再继续

### 第 2 步：分析图片并挑选产品图

先读取 `要求.yaml` 和 `配置.yaml`，再查看所有输入图片，并分析：
1. **客户要求**：从 `要求.yaml` 读取 client、season、style、folder_requirements，这些是客户的核心需求
2. **服装信息**：品类、颜色、面料、廓形、图案、关键设计点
3. **模特信息**：五官、身材、发型、整体气质
4. **风格参考图**：色盘、影调、光线质量、构图习惯、环境层次、编辑感
5. **客户细节图**：如果客户特意拍了腰部、领口、袖口、面料纹理等区域，这些位置必须作为近景重点优先呈现
6. **配置要求**：读取 `配置.yaml` 中的 `selected_images`、`model_image` 与 `custom_requirements`，它们优先级高于你自己临时猜测

优先使用 `配置.yaml` 里选定的服装图发送给 API。默认应尽量精简，但当产品存在明显的**正面 / 背面 / 下装全貌 / 腰头或抽绳细节**拆分时，可以保留 3-6 张结构图，并通过 `selected_image_notes` 与场景级 `source_image_indexes` 明确它们的职责。不要无区别地把目录里的全部图片直接送进去。

这里的“查看所有输入图片”是强制要求，不允许只读目录名、要求文本或沿用上一款的推断。写场景前必须真的看过这款衣服的源图，确认颜色、面料、廓形、长度、前后结构、下装形态、细节位置与卖点。

#### 装饰元素清单（强制）

看完源图后，**必须先列出这件衣服所有可辨认的装饰元素**，每个元素标明：

- **名称**：如珍珠链、金属钉珠、装饰绳、宝石纽扣、绑带、蝴蝶结、流苏等
- **位置**：仅正面 / 仅背面 / 两侧都有 / 腰部 / 领口 / 袖口 / 下摆等
- **形态**：连续链状 vs 散珠、编织绳 vs 布带、暗扣 vs 明扣等

示例：
```
装饰元素清单：
1. 珍珠链 → 仅正面领口沿边一圈，连续链状，背面无
2. 蓝色宝石纽扣 → 正面前中从腰线到裙摆，一列排列
3. 自裁腰带 → 正面系蝴蝶结，背面自然飘带
4. 泡泡袖翻折边 → 两侧袖口
```

这份清单将直接指导后续 `clothing_description`、`required_details`、`back_only_details` 和 `forbidden_details` 的编写。清单中标注"仅正面"的元素，**严禁**在背面场景的 `required_details` 或 `back_only_details` 中出现；反之亦然。

先用中文向用户汇报分析结果（包含装饰元素清单），并在继续前确认。

### 第 2.5 步：同季同风格批量避重检查（批量任务强制）

当用户要连续生成同一季、同一风格下的多款产品（例如 `2026_春夏_第二批`）时，设计当前产品的 `scenes.json` 前，必须先做一次同季同风格的避重检查：

1. 在当前产品同级目录中，找到同一个 `season + 风格名` 下最近的 4-8 个兄弟产品，优先读取其中**已经正式生成过**的 `生成/batch_*/scenes.json`
2. 提炼这些兄弟产品里已经高频出现的：
   - **hero 场景家族**：前 1-10 张全身镜头使用的主环境骨架
   - **微场景**：例如“温室花房入口”“弧墙楼梯转角”“稻田田埂”“精品门店外摆左侧位”这种具体落点
   - **道具组合**：例如花束+藤编篮、结构包+弧墙、草帽+编织篮、咖啡杯+橱窗
3. 当前产品必须主动避开：
   - 与最近 3 个兄弟款重复相同的前 10 张全身场景骨架顺序
   - 相邻 2 款重复同一个微场景
   - 同一套道具组合连续 3 款重复
4. 如果某个主场景因版型、材质或客户要求必须沿用，至少把**其余 3 个全身镜头 + 2 个近景镜头**轮换到未高频使用的环境与道具

⚠️ 重点：这里要求的是**同风格批量防重**，不是单款内部随机。即使当前衣服本身适合某个风格锚点，也不能把整个风格批次都压成同一套场景骨架。

### 第 3 步：设计 10 个场景并写入 scenes.json

基于本文件规则，为该服装**设计 10 个互不重复的场景**，并写入批次目录中的 `scenes.json`。

#### scenes.json 生成禁令

- 严禁使用任何脚本直接或间接产出 `scenes.json`，包括批量脚本、一次性辅助脚本、批量改字脚本和模板填充脚本。
- 仓库中不应保留用于产出 `scenes.json` 的 `.mjs` / `.js` / `.py` 辅助脚本；如果发现旧脚本，默认删除，不再使用。
- 严禁把一整套 20 个场景从上一款复制过来后只改产品编码、颜色、面料名或少量名词。
- 严禁同风格多款服装共用同一组“正面全身/背面全身/动态/氛围/互动”的固定背景模板。
- 如果目录中已经存在一个**尚未正式出图**的旧 `scenes.json`，它只能视为废弃草稿；必须重新看图后重写，不能直接沿用。
- 如果该产品已经有正式生成结果，默认不要重写 `scenes.json`，除非用户明确要求返工。

#### scenes.json 设计要求

- 每个场景都要写出这款衣服独有的卖点，不允许出现“适用于任意上衣/套装/裙装”的空泛场景话术。
- 场景差异必须来自这款衣服本身：例如版型、长度、材质、正反结构、是否有口袋、领口/袖型/腰头/后背细节、鞋型适配、适合的动作和环境。
- 近景镜头要跟随客户源图里的真实细节位置设计，不允许用一套通用细节拼图模板覆盖所有款。
- 如果当前任务属于同季同风格批量生成，当前 20 个场景不仅要区别于这件衣服本身，还必须区别于最近兄弟 SKU 已经用过的高频场景骨架。
- 同一产品内部仍保持固定的 20 镜头职责，但**环境、动作、道具、微场景、光线状态**必须轮换，不能把每件衣服都写成同一套“花房入口 / 楼梯 / 稻田 / 咖啡馆 / 藤蔓墙”顺序。
- 对于批量任务中的 `scene_01 ~ scene_18`，`pose / lighting / background / model_direction / color_tone` 这 5 个字段必须全部写满，不能只剩 `scene_family + micro_location + full_prompt`。如果这 5 个字段大量缺失，说明场景控制不完整，视为不合格的 `scenes.json`。
- `scene_family` 的变化不能只是换名字但仍停留在同一个视觉母体里。若多个 SKU 都是同一种暖墙、同一种半户外、同一种树影+台阶语汇，只是微场景改字，仍然属于重复。
- 对 `艺术建筑 (E)` 的同季批量任务，必须先从 `references/styles/style-e-architectural-editorial.md` 的 **20 个具体建筑场景方案** 中选择 1 个，作为当前 SKU 的唯一主场景世界。
- 对 `艺术建筑 (E)` 的 `scene_01 ~ scene_18`：必须始终停留在同一个建筑世界中，只允许切换这个建筑内部或同一建筑群内**逻辑连通**的 2-3 个 micro-location，不允许跨到另一个建筑方案去“拼盘”。
- 对 `艺术建筑 (E)` 的相邻兄弟 SKU：不应共享同一个建筑场景方案编号；连续批量任务优先按 20 个方案库轮转。
- 写完 `scenes.json` 前，先自检一次：如果把产品编码和服装名遮掉，这 20 个场景仍然能一眼看出是为这件衣服定制的，才算合格。
- **装饰元素交叉验证**：逐一核对装饰元素清单中的每个元素，确认：
  - 标注"仅正面"的元素没有出现在任何背面场景的 `required_details` 或 `back_only_details` 中
  - 标注"仅背面"的元素没有出现在任何正面场景的 `front_required_details` 中
  - 所有装饰元素至少在一个场景的 `required_details` 中被要求展示
  - `clothing_description` 中对每个装饰元素的位置描述与清单一致
- **forbidden_details 针对性检查**：确认每个场景的 `forbidden_details` 包含至少 2-3 条**该衣服特有的**易错项（不能只有通用模板项）

**输出路径**：优先使用 `客户/{客户名}/{season}/{风格名}/{产品编码}/生成/batch_NN/scenes.json`。若遇到历史项目，可兼容旧路径 `客户/{客户名}/产品 2/{风格名}/{产品编码}/生成/batch_NN/scenes.json`

系统应自动检测下一个批次号（`batch_01`、`batch_02` ...）。

如果存在多个客户文件夹、参考批次或群图文件夹，始终优先使用**最新文件夹 / 最新版本**，并先阅读文件夹级备注与 `配置.yaml`。新的群图、备注、配置、以及最新文件夹内容优先级最高，旧文件夹只能作为补充参考。

### 第 4 步：渲染图片

`scenes.json` 必须是完全自包含的：里面包含所有图片路径、输出目录和完整 prompt。渲染脚本只负责读取并执行：

```bash
cd /Users/orion/SudoAI/jiaopiantai
deno run --allow-read --allow-write --allow-env --allow-net \
  .agent/skills/styled-clothing-shoot/scripts/render.ts \
  -j "客户/.../{season}/{风格名}/{产品编码}/生成/batch_NN/scenes.json"
```

只重渲染单个场景：

```bash
deno run --allow-read --allow-write --allow-env --allow-net \
  .agent/skills/styled-clothing-shoot/scripts/render.ts \
  -j "客户/.../{season}/{风格名}/{产品编码}/生成/batch_NN/scenes.json" \
  --scene 3
```

### 第 5 步：复盘与迭代

图片生成完成后，检查质量并询问用户是否需要调整。若只需重做某一个场景，直接修改 `scenes.json` 中对应场景，再单独渲染该场景。

### `2026_春夏_第二批` 批量任务提示

当前仓库中 `客户/花姐&寅寅/2026_春夏_第二批/` 已存在多组同风格产品，主要集中在：
- `田园自然`
- `艺术建筑`
- `大自然`

后续批量生成这些目录时，必须把“读取兄弟 SKU 的既有批次并主动避重”当成默认动作，而不是额外可选项。尤其不要把：
- 田园自然都写成“花房门口 + 花园小路 + 草地坐姿 + 藤蔓墙”
- 艺术建筑都写成“弧墙门洞 + 楼梯 + 长廊 + 台阶”
- 大自然都写成“普通草地 + 普通稻田 + 压暗黄昏 + 山路回头”

这些都属于**同风格骨架重复**，必须主动轮换。

位于 `客户/{客户名}/风格参考/`：
- `风格一_海边艺术/`：重点学习海岸氛围、暖色调、岩石/浪花环境，以及巴厘岛式热带海边、海边餐厅、海边艺术馆、礁石等更丰富的海景叙事
- `风格二_田园自然/`：重点学习田园柔和感、自然暖光、花园/庭院/藤蔓/白纱设定，以及更高级的庄园花园、温室花房、露台场景
- `风格三_都市松弛/`：重点学习城市生活感、室内外混合、设计空间与自然松弛状态，可延展到精品咖啡馆、画廊、美术馆咖啡区
- `风格四_现代科技/`：重点学习未来都市女装广告感、玻璃/金属/亚克力材质、柔光灯带、展厅式空间、低饱和冷调配色、高级精致而不过度概念化的科技氛围
- `风格五_艺术建筑/`：重点学习暖米色矿物墙面、几何建筑线条、厚墙门洞、平墙树影、方柱廊道、宽石台阶、低台庭院、内廊看向亮庭院等多种建筑家族，不要再压成单一“弧形楼梯/门洞/长廊”模板
- `风格六_室外街拍/`：重点学习门店外景、街边咖啡馆、露天餐厅、橱窗与路面关系、漂亮建筑与街角、松弛抓拍姿态，以及阳光 / 树荫 / 明亮阴天的真实街头光线；同时优先吸收“干净街角、少人背景、简洁外立面、低视觉噪音”的参考特征
- `风格七_大自然/`：重点学习芦苇荡、花海、草原坡地、梯田观景点、湿地栈道、山野露台等更精致的自然环境，以及有阳光、有空气感的明亮山野影调；不要学习阴暗灰闷或被虚化糊掉的自然背景

⚠️ 这些图片是**情绪参考板**，要学习的是感觉（色盘、影调、光线质量），不是原图构图照搬。每件服装都需要重新设计专属场景。

## 20 镜头框架

每个产品固定生成 **20 张图**：**10 张远景/全身 + 10 张近景/局部**，严格保持 **50% 远景、50% 近景**。不要让结果偏成大多数都是全身图，也不要把后 10 张都做成重复的小裁切。

### 全身镜头（10 张 — `framing: "full_body"`）

| # | 镜头名 | 说明 |
|---|--------|------|
| 1 | 正面全身 | 全身正面视角。姿态要自然带动作，例如整理配饰、向前迈步、轻微转身。重点展示服装正面穿着效果。 |
| 2 | 背面全身 | 全身背面视角。可以走开、回头、迎风而立，体现背面轮廓与线条，画面要有故事感。 |
| 3 | 全身动态 | 动态全身姿态，如行走、转身、与环境互动。重点表现服装垂坠感和动态表现。 |
| 4 | 全身氛围 | 全身生活方式镜头，如坐姿、倚靠、半蹲、放松停留。体现服装在自然状态下的上身效果。 |
| 5 | 全身环境互动 | 与环境发生更强互动的全身镜头，如扶栏杆、穿过门洞、靠墙、踩水边、贴近场景道具。场景要与前 4 张不同。 |
| 6 | 全身侧身 / 45 度 | 用 45 度或轻侧身视角展示廓形、肩线和衣摆走势，避免又变成正面站姿复刻。 |
| 7 | 全身坐姿 Hero | 坐在风格化道具或空间节点上的完整造型图，重点看上身比例、腿部姿态和环境关系。 |
| 8 | 全身门洞 / 楼梯 / 长廊 | 借助建筑或空间结构做纵深构图，适合体现层次和系列感。 |
| 9 | 全身前景遮挡 | 允许用植物、门框、栏杆、纱帘、树影等做前景，形成更强编辑感，但服装主体必须清楚。 |
| 10 | 全身远景 Hero | 全身入画但环境占比更大的一张英雄图，用于拉开系列节奏；环境可更完整，但服装仍是第一视觉中心。 |

### 近景 / 局部镜头（10 张 — `framing: "close_up"`）

| # | 镜头名 | 说明 |
|---|--------|------|
| 11 | 上半身近景正面 | 只拍上半身，裁切到腰部或胸部附近。重点展示领口、肩线、袖型、面料纹理。可加入手触面料或配饰动作。 |
| 12 | 上半身近景侧面 | 用侧面或 3/4 侧脸近景展示肩颈、袖型、层次和情绪状态，和第 11 张拉开视角差。 |
| 13 | 领口 / 肩线细节 | 聚焦领口、翻领、肩线、胸前结构、胸前装饰等最核心卖点。 |
| 14 | 袖口 / 前片细节 | 聚焦袖口、前襟、纽扣、钉珠、抽绳、胸前纹理等局部。 |
| 15 | 腰部 / 口袋 / 腰头细节 | 聚焦腰线、口袋、腰头、抽绳、绑带、褶皱等中段结构。 |
| 16 | 下半身近景 | 只拍下半身，重点展示裤型/裙摆、下摆线条、面料垂感，以及步态或站姿下的廓形。 |
| 17 | 鞋履与下摆近景 | 聚焦裤脚/裙摆与鞋履之间的关系。鞋履搭配以最新群图/参考图为基准，可根据风格选择高跟鞋、踝靴、乐福鞋、长靴或简洁凉鞋，不要机械统一成一种鞋型。 |
| 18 | 背面局部细节 | 聚焦后领、背部线条、后腰、后摆、背面装饰等，避免背面只靠全身图呈现。 |
| 19 | 多角度细节拼图 | 默认做 **2+1 / 1+2 / 双列细节拼图**，把 3-4 个最值得卖的细节放进同一张图，例如领口 / 袖口 / 腰部 / 口袋 / 面料纹理。面板之间用细白线或微妙留白分隔，整体像高端 lookbook 细节页。 |
| 20 | 正反面拼图 | 默认做**双面板或三联式正反面销售拼图**，至少同时展示正面完整造型与背面完整造型；若做三联，可增加一个关键局部。所有面板必须是同一模特、同一场景体系、同一影调。 |

⚠️ 近景规则：
- 不允许模特全身完整出镜，只能是身体局部
- 必须清晰呈现服装细节（纹理、编织、色彩层次）
- 背景景深要自然克制：默认可以有轻微软化，但禁止明显的软件式大虚化、奶油化假景深；若为 `大自然 (G)`，必须保留芦苇、花海、草坡、草原、梯田等环境纹理与空间层次，不能把自然背景糊成一片
- 近景仍需有编辑感，不是平铺式产品裁切图
- 第 13-20 张要整体覆盖领口、袖口、腰部、口袋、下摆、鞋履、背面结构和关键细节等销售重点
- 第 19 张默认是**多角度细节拼图**，第 20 张默认是**正反面拼图**
- 如果客户提供了最新群图 / 拼图示意 / 参考版式，拼图场景优先跟随该版式，不要自行忽略
- 正反面拼图虽然归在第 20 张近景/局部组里管理，但它是**版式型销售图**：允许面板内出现全身，只要整张图明确是拼图销售页而不是单张普通全身图

## 场景设计规则

### 自然感（最高优先级）

🚫 **绝对禁止**：
- 像证件照一样僵硬直站
- 夸张扭曲的模特姿势
- 正面僵硬直视镜头
- 棚拍肖像照式的僵硬表现

✅ **必须做到**：
- 每个姿态都像“被自然抓拍到的瞬间”，例如风吹发丝、半转身、行走步态、整理衣物
- 视线尽量看向远方、地面、道具或侧脸方向，**多数情况下不要直视镜头**
- 身体重心自然，一条腿微弯，肩膀放松，整体有呼吸感
- 头发和衣服要有自然风感或动态感
- 整体质感要像高端生活方式杂志抓拍，而不是棚拍摆拍

### 影调与调色（必须贴合参考图）

每个场景都要明确写出：
- **色温**：暖调还是冷调？户外光线必须匹配所选风格，例如海边艺术 / 艺术建筑偏清晰日照和硬影，室外街拍可接受树荫或明亮阴天，大自然可接受明亮多云或薄暮田野
- **反差**：高反差戏剧感，还是低反差胶片感
- **饱和度**：浓郁电影感，还是轻盈通透感
- **光线质量**：逆光轮廓、侧光塑形、还是柔和漫射
- **后期气质**：胶片感、电影感、干净清透感等

每个场景里的 `color_tone` 必须写得具体，例如：`黄金时刻暖调，逆光轮廓光，低对比度胶片质感`，不要只写“自然光”“柔和光线”这种空泛描述。

### 背景复杂度（硬性要求）

每个背景都必须达到杂志级别的层次：
- **前景**：虚化植物、岩石边缘、建筑线条、布料或道具等景深元素
- **中景**：模特和服装是视觉中心
- **背景**：远处海面、山体、建筑群、室内空间延伸等环境叙事
- **氛围**：光影、色调、天气、时间、环境纹理共同服务画面

🚫 禁止：纯色墙、大面积空沙滩、单平面背景、没有层次的空画面  
✅ 必须：背景本身能讲故事，具备环境叙事性

### 背景人群与场景重复控制（硬性要求）

- 默认不要出现背景路人、游客、排队顾客、餐厅散客、路边行人或其他抢镜人物；除非用户明确要求群像，否则画面中只能有主模特一人。
- 室外街拍 / 都市松弛不能 20 张图都套同一种街边模板。要主动拉开差异，随机覆盖咖啡馆、漂亮露天餐厅、漂亮建筑立面、街角、橱窗、画廊入口、艺术馆外部等至少 4 类环境。
- 室外街拍优先使用可控且干净的外景：精品门店外立面、安静街角、非高峰时段的咖啡馆外摆、简洁建筑转角、低车流低人流的人行道。不要选择人群密集的商业街、杂乱招牌墙、停满车的路边、拥挤斑马线或背景信息过多的游客区。
- 室外街拍构图要主动避开杂乱背景：减少远处行人、车流、路障、杂物桶、密集招牌、电线杆和凌乱反光；必要时用更近的机位、浅景深、建筑转角遮挡、门廊或树影前景来清理背景。
- 海边艺术不能只停留在“空沙滩 + 海平线”。要主动在礁石海岸、潮汐池、巴厘岛式热带海边、海边餐厅、热带棕榈屋、海边艺术馆、海岸步道等场景中做变化。
- 田园自然 / 艺术建筑 / 室外街拍都要把“高级感”写进背景与光线：材质、建筑线条、植物修剪感、桌椅器物、树影层次、餐具或空间细节都要有杂志感，而不是廉价游客照背景。

### 户外天气、鞋履、配饰与造型硬要求

- 户外天气必须跟随风格参考，不要一刀切。A / E 优先晴天直射光、黄金时刻和清晰阴影；B 优先明亮暖光、柔和日照和林荫斑驳光；F 允许午后斜阳、街边树荫或明亮阴天；G 允许晴天山野、明亮多云或薄暮田野。无论哪种外景，都不要压成脏灰、发闷、无层次的天气。
- 鞋履必须以最新群图 / 参考图和风格气质为基准。A / B / D 可优先高跟鞋或更精致鞋型；E / F / G 可根据造型使用踝靴、乐福鞋、长靴、平底凉鞋或高跟鞋。不要把所有下装机械统一成高跟鞋。
- 配饰要主动加丰富，帮助提升商业销售感：耳环、帽子、墨镜、手提包、花束、咖啡杯，以及其他适合场景的配饰都可以加入。
- 如果是连衣裙 / 半裙 / 裤装，默认按有口袋成衣处理；场景描述和细节图里要明确口袋结构，至少随机安排 1-2 张自然手插口袋或拇指勾住口袋边的照片，但不要每张都重复同一动作。
- 所有服装默认按有内衬、成片不透处理。除非源图和客户要求明确指定透明设计，否则禁止把裙身、腿部、内衣轮廓或肤色透出成薄纱 / 透视效果。

### 模特身份一致性

10 个场景必须都是**同一个模特**。每个场景都要写 `model_direction`，说明：
- 该场景中的表情和情绪
- 发型如何配合场景
- 身体语言、能量状态、与环境的互动方式

不要重新设计人脸，也不要生成新的身份。

### 画幅比例

所有场景统一为 **3:4 竖版构图**。

### 字段说明

- `metadata.clothing_images`：来自产品目录根部、并由 `配置.yaml` 选定的服装图；可根据产品复杂度保留 2-6 张，但必须有明确分工
- `metadata.source_image_notes`：可选，按顺序说明每张源图的角色（如“正面整体”“背面整体”“裙裤平铺”），用于防止模型混淆正反面或细节归属
- `metadata.product_config`：产品目录下的 `配置.yaml` 路径。建议始终填写；若 `clothing_images` 或 `model_image` 缺失，渲染脚本可从这里读取 `selected_images` 与 `model_image`
- `metadata.requirements_config`：产品目录下的 `要求.yaml` 路径，包含 client、season、style、folder_requirements 等客户核心要求
- `metadata.batch_diversity_context`：**批量任务推荐填写**。记录当前检查了哪些兄弟 SKU、当前要避开的高频场景组合，便于后续继续接力时保持多样性
- `metadata.style_scenario`：可选但强烈推荐。对于 `艺术建筑 (E)` 这类“先选场景方案再展开镜头”的风格，建议明确写入选中的方案编号与名称，例如 `E07 修道院式回廊博物馆`
- `metadata.client`：从 `要求.yaml` 读取的客户名
- `metadata.season`：从 `要求.yaml` 读取的年份季节
- `metadata.style`：从 `要求.yaml` 读取的风格名
- `metadata.folder_requirements`：从 `要求.yaml` 读取的具体要求，场景设计时必须体现
- `metadata.model_image`：模特参考图路径（相对项目根目录）。可显式填写；若留空，则优先从 `配置.yaml` 回填
- `metadata.output_dir`：`scene_01~20.png` 的输出目录，也是 `scenes.json` 所在目录
- `framing`：第 1-10 张使用 `full_body`，第 11-20 张使用 `close_up`。**第 19-20 张即使是拼图，也继续写 `close_up`，靠 `crop_focus` 和 `full_prompt` 明确多面板布局。**
- `scene_family`：**批量任务推荐填写**，用于标记当前场景属于哪一类风格锚点，如“温室 / 花房”“弧墙 / 长廊”“稻田 / 田埂”
- `micro_location`：**批量任务推荐填写**，写到具体落点，例如“花房入口右侧白纱门洞”“弧形楼梯第二个平台”“稻田木栈道转角”
- `diversity_reason`：**批量任务推荐填写**，简述本镜头相对近期兄弟 SKU 的避重理由
- `pose`：**批量任务必填**。不能只写“站立 / 行走 / 坐姿”，必须写成有身体重心、手部动作、视线方向和故事感的姿态描述。
- `lighting`：**批量任务必填**。必须写清光线方向、质量、时间感与反射来源，不能只写“自然光”。
- `background`：**批量任务必填**。至少交代前景 / 中景 / 远景层次，以及当前风格真正成立的环境材质与空间支点。
- `model_direction`：**批量任务必填**。写清模特情绪、表情与身体状态，避免所有款都变成同样的站姿气质。
- `color_tone`：**批量任务必填**。必须写成可执行的影调说明，而不是“高级感 / 杂志感”这种空话。
- `crop_focus`：全身图留空；近景图里要明确描述展示的是哪个区域 / 配饰 / 细节。**拼图场景必须明确每个面板各自拍什么、面板如何分隔。**
- `source_image_indexes`：**推荐**（多源图时强烈推荐），按 `metadata.clothing_images` 的 1-based 序号选择当前场景要发送的源图。**正面场景只喂正面图/下装图/正面细节，背面场景只喂背面图/下装图**，避免模型把正面装饰画到背面或反之。当源图 ≥3 张且有明确正反面分工时，必须使用此字段
- `render_goal`：可选，`validation` 表示该镜头优先验证服装结构，`final` 表示正式成片；渲染脚本会把它转成更硬性的提示
- `required_details`：**推荐**，该镜头必须保留的通用细节。应从装饰元素清单中挑选与当前镜头相关的元素，具体到位置和形态
- `front_required_details`：**推荐**（正面场景必填），只要镜头涉及正面，这些细节必须清楚出现。只能包含装饰元素清单中标注为"正面"或"两侧"的元素
- `back_only_details`：**推荐**（背面场景必填），只允许出现在背面或背面面板中的结构。只能包含装饰元素清单中标注为"背面"或"两侧"的元素
- `bottom_required_details`：**推荐**（下装可见场景必填），下装必须被保留的结构细节，适合裙裤、腰头、抽绳、裤腿分离感、装饰绳等
- `forbidden_details`：**推荐**，列出模型最容易犯的错误，渲染时会附加到 prompt。**必须包含该衣服特有的易错点**，不能只用通用模板。典型的衣服特有 forbidden 包括：
  - 正面场景：禁止出现仅属于背面的结构（如"背面绑带不应出现在正面"）
  - 背面场景：禁止出现仅属于正面的装饰（如"正面珍珠链不应出现在背面"）
  - 将某装饰渲染成错误形态（如"把珍珠链渲染成散珠""把装饰绳渲染成普通缝线"）
  - 遗漏关键装饰元素（如"裤腿缺少装饰绳""领口缺少钉珠"）
- `seed`：每个场景都用不同的 7 位随机数
- `full_prompt`：必须是**完整可直接发送**的 prompt
- 所有文本字段必须使用简体中文

### clothing_description 编写规范

`clothing_description` 是告诉 AI "这件衣服长什么样"的核心字段，写错会直接导致生成错误。必须遵守以下规则：

1. **来源必须是源图，不是猜测**：每一条描述都必须能在源图中找到对应的视觉证据。如果源图看不清某个细节，写"根据源图难以确认"而不是编造。
2. **装饰元素位置必须精确到正/反面**：不能写"领口有珍珠链"（歧义），必须写"仅正面领口沿边缀一圈珍珠链，背面领口无装饰"。
3. **区分前后差异**：如果正面和背面的结构不同（如前排扣后拉链、前钉珠后系带），必须分别描述。
4. **装饰形态要具体**：不能只写"珠子装饰"（AI 会自由发挥），必须写"连续珍珠链"或"单颗金属钉珠均匀排列"。
5. **所有 20 个场景共用同一个 `clothing_description`**，因此描述必须同时覆盖正面和背面的所有特征，并明确标注哪些属于正面、哪些属于背面。

#### 常见 clothing_description 错误（必须避免）

| ❌ 错误写法 | ✅ 正确写法 | 后果 |
|------------|-----------|------|
| "领口有珍珠链" | "仅正面领口沿边缀一圈连续珍珠链，背面领口无装饰" | AI 在背面也画珍珠链 |
| "裤子有装饰" | "裤腿两侧各有一条垂坠装饰绳，绳端有珠饰" | AI 遗漏装饰绳 |
| "扣子装饰" | "正面前中排列5颗珍珠扣，第一颗在锁骨位置" | AI 生成错误扣子类型 |
| "面料有图案" | "浅冰蓝底色上覆盖抽象笔触风格海洋印花，以深藏蓝海星形态和粉色曲线交织" | AI 生成错误图案 |
| "有绑带设计" | "后领口有布质系带，正面无绑带；腰部自裁腰带正面系蝴蝶结" | AI 在正面也画绑带 |

### 关键镜头先验收

当产品容易出现结构混淆（例如正反面串位、裙裤变半裙、内搭消失）时，推荐先只重渲染并验收以下 5 张关键镜头，再放量整批：

1. `scene_01`：正面全身，检查正面主结构
2. `scene_02`：背面全身，检查背面线条与后片结构
3. `scene_11`：上半身近景，检查领口/袖型/内搭
4. `scene_16`：下半身近景，检查裙裤结构与腰头
5. `scene_20`：正反面拼图，检查正反面分工是否清楚

## Prompt 组装规则

每个场景的 `full_prompt` 都必须是一个**完整、自包含**的艺术指导 prompt。渲染脚本不会再做二次加工，而是直接发送给 Gemini API。

模板如下：

```
{brand_anchor}

图像角色规则：产品源图仅用于锁定服装本身。模特参考图用于锁定人物身份。最终人物必须与模特参考图是同一个人，保持一致的五官、脸型、发型、体型和气质。

模特一致性规则：这是同一个模特的系列拍摄。所有场景中的人物必须看起来是同一个人，只是在不同场景中拍摄。可以调整妆容和表情来适配场景氛围，但面部身份不得改变。

姿态规则：模特的姿态必须自然松弛，像被抓拍到的瞬间，不要像拍肖像照那样僵硬。视线自然（看远方、侧脸、低头等），身体重心自然分配，头发和衣物有自然的风感和动态。整体感觉是高端生活方式杂志的自然抓拍，不是棚拍肖像。

影调规则：画面色调必须有电影质感和高级感。要有明确的色温倾向（暖调或冷调）、适当的对比度、和丰富的光影层次。避免平淡无影调的普通照片感。光线要有方向感和质量感。

商业硬规则：默认不要出现背景路人或拥挤人群。若服装属于连衣裙、半裙或裤装，要把口袋结构清楚做出来，并在合适镜头里安排自然手插口袋 / 勾口袋边动作。所有服装默认有内衬、成片不透，除非客户明确要求透明设计，否则不能出现透视、透腿、透出内衣或肤色的效果。

构图规则：{framing_rule}

本场景模特状态：{model_direction}

服装描述：{clothing_description}

拍摄风格：{style_name}

{scene_prompt — 融合 pose / lighting / background / color_tone 的 100-150 字完整场景描述}

Output image resolution: {resolution}. Ultra high resolution, sharp details.

Negative: {negative}
```

其中 `{framing_rule}` 为：
- **full_body**：`3:4竖版构图，人物全身入画，背景丰富有层次，明确呈现远景环境叙事。`
- **close_up**：`3:4竖版构图，近景/局部构图。{crop_focus}。默认情况下画面只展示模特身体的一部分（上半身、下半身、鞋履、口袋或配饰区域），不要展示全身。若该场景是第 19-20 张拼图，则必须明确写出多面板布局、每个面板的拍摄重点和面板间的细白线/留白分隔；其中第 20 张正反面拼图允许面板内出现全身，但整张图必须明确是拼图销售页，而不是单张普通全身图。服装细节必须清晰可见（面料纹理、编织方式、色彩层次、配饰搭配）。背景适度虚化，视觉焦点集中在衣服细节和造型卖点上。仍需自然的姿态和光影，不是平铺产品图。`

`brand_anchor` 和 `negative` 使用标准值，并在整个项目中保持一致。

⚠️ `full_prompt` 必须 100% 完整可直接发送。渲染脚本会将 `full_prompt` 作为主体，并自动追加 `required_details`、`front_required_details`、`back_only_details`、`bottom_required_details` 和 `forbidden_details` 作为额外约束块发送给 Gemini API。因此 `full_prompt` 应包含完整的场景描述，而细节约束通过上述字段单独管理。

## 约束与守则

- 所有场景文案必须使用简体中文
- 必须严格保留服装颜色、材质、logo、图案、版型和结构
- 每个场景姿态都必须不同，不能重复
- 20 张图中的模特身份必须一致
- 对同季同风格批量任务，必须先回看最近兄弟 SKU 的既有批次，避免重复同一套场景骨架
- 相邻 SKU 不允许共享相同的前 10 张 hero 场景排序；即使风格相同，也必须轮换微场景、道具和主环境
- 默认清场，不要生成背景路人、游客、服务员、逛街人群或其他抢镜人物
- 要主动补足有商业价值的配饰与鞋履，尤其是耳环、帽子、墨镜、手提包、花束、咖啡杯，以及符合风格的高跟鞋、踝靴、乐福鞋、长靴等
- 街景、田园、海景、艺术馆和建筑空间都要有高级感；不要反复套用同一条街、同一面墙、同一片空海滩
- 连衣裙、半裙、裤装默认按带口袋款处理，至少 1-2 张图要自然展示手插口袋或口袋边互动
- 所有服装默认有内衬、最终成片不透；禁止擅自做成半透明薄纱或透视装
- 人脸不得重设计，必须保留模特参考图中的身份
- `styled-clothing-shoot/` 内部脚本必须保持自包含，不得依赖兄弟 skill 或外层 skill 的运行时代码
- 必须先读取产品目录下的 `要求.yaml`，获取 client、season、style、folder_requirements，并在场景设计中体现客户的核心要求
- 必须先读取产品目录下的 `配置.yaml`，并把其中的 `selected_images`、`model_image` 与 `custom_requirements` 体现在 scenes 设计中
- 如果 `配置.yaml` 已知某些源图分别对应正面、背面、平铺或补充细节，优先写入 `selected_image_notes` 并在 prompt 中强化这些图序语义，避免前后结构串位

## 常见生成错误与防御策略

以下是经过人工评审确认的高频生成错误，设计 `scenes.json` 时必须针对性防御：

### 1. 正反面装饰串位（占比最高）

**现象**：正面专属的装饰（如珍珠链、钉珠）出现在背面，或背面专属的结构（如系带、拉链）出现在正面。

**防御**：
- `clothing_description` 中明确写"仅正面"/"仅背面"
- 背面场景的 `forbidden_details` 加入 "正面XX装饰不应出现在背面"
- 正面场景的 `forbidden_details` 加入 "背面XX结构不应出现在正面"
- 使用 `source_image_indexes` 避免喂入无关角度的源图

### 2. 装饰元素遗漏（第二高频）

**现象**：源图中明显的装饰绳、流苏、钉珠、特殊扣子等在生成图中消失。

**防御**：
- 在 `required_details` 中逐一列出当前场景可见的装饰元素
- 在 `forbidden_details` 中加入 "遗漏XX装饰" / "XX处缺少装饰绳"
- `clothing_description` 中对装饰绳、流苏等容易被忽略的元素要着重描述形态和位置

### 3. 装饰形态错误

**现象**：珍珠链被渲染成散珠、装饰绳被渲染成普通缝线、宝石纽扣被渲染成普通纽扣。

**防御**：
- 描述时使用精确形态词："连续珍珠链"而非"珠子"、"垂坠绳饰带珠尖"而非"装饰绳"
- `forbidden_details` 加入形态纠正："把珍珠链渲染成散珠"、"把装饰绳渲染成普通缝线"

### 4. 版型/结构错误

**现象**：领口形状变化、口袋消失或变形、裙裤变半裙、背后裁剪线不对。

**防御**：
- `clothing_description` 中精确描述领口类型（V领/圆领/方领/翻领）
- 有口袋的衣服必须在 `required_details` 写明口袋类型和位置
- `forbidden_details` 写明常见的版型混淆（如"把裙裤生成成半裙"）

### 5. 花纹/图案不匹配

**现象**：抽象印花被简化成规则碎花、复杂图案的色彩组合被改变。

**防御**：
- `clothing_description` 中详细描述图案的具体元素和色彩组合
- `forbidden_details` 加入 "把XX印花生成成YY"
- 对于复杂图案，在 `required_details` 中强调关键图案元素

### 6. 手部畸形（AI 模型限制）

**现象**：手指数量异常、手部姿势不自然。

**防御**：
- `forbidden_details` 始终包含 "畸形手"、"额外肢体"
- 避免设计需要复杂手部动作的场景（如手指交叉、多手指特写）
- 近景场景适当让手部位于画面边缘或被遮挡
- 若无用户明确禁止，**20 张图里必须包含 2 张拼图版式**：第 19 张多角度细节拼图，第 20 张正反面拼图
- 拼图版式优先参考**最新群图 / 拼图 / 参考批次**中的版式语言；若已有明确 2+1、1+2、左右双联等结构，应优先沿用而不是重新发明
- 不要再假设存在 `原图/` 子文件夹；产品图默认就在产品目录根部
- 如果用户提供的是平铺图，要提醒对方：服装照片越清晰，AI 结果越稳定
