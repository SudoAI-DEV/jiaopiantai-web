## Context

当前审核弹窗由 `src/components/review/image-review-modal.tsx` 管理两层交互：主审核图的左右切换，以及原图覆盖层的放大查看。原图覆盖层当前只保存 `sourcePreviewUrl`，因此只能打开单张原图，无法基于原图顺序做键盘切换。与此同时，弹窗操作区通过 `isPending` 将审核按钮限制为仅 `pending` 图片可操作，导致 `approved` 和 `rejected` 图片无法在弹窗内直接修正结论。

现有后端接口 `PATCH /api/admin/images/[id]/review` 已接受 `pending / approved / rejected / regenerate` 状态，并且对非 `rejected` 状态会清空 `rejectionReason`。因此这次变更主要是前端交互与状态流转约束的调整，不需要新增 API 或数据库字段。

## Goals / Non-Goals

**Goals:**
- 让原图放大覆盖层支持按原图顺序使用 `↑ / ↓` 键切换其他原图
- 在原图切换时重置放大缩放和拖拽位置，避免沿用上一张图的平移状态
- 允许管理员在审核弹窗中将 `approved` 改为 `rejected`，以及将 `rejected` 改为 `approved`
- 保持驳回理由的校验与清理规则一致：驳回必须填写理由，改回非驳回时清空理由
- 保持现有 pending 图片的连续审核效率，不因为新增纠错能力破坏原有自动翻页体验

**Non-Goals:**
- 不新增审核状态，也不改变 `pending / approved / rejected / regenerate` 的业务语义
- 不为审核网格页新增已审核图片的悬浮快捷按钮，纠错入口仍以审核弹窗为主
- 不改造批量审核流程
- 不引入新的图片预览或快捷键第三方库

## Decisions

### 1. 原图预览状态从 URL 改为索引驱动

原图覆盖层的状态由 `sourcePreviewUrl: string | null` 改为 `sourcePreviewIndex: number | null`，当前展示的原图从 `sourceImages[sourcePreviewIndex]` 派生。

这样做的原因：
- `↑ / ↓` 导航天然依赖列表顺序，索引比 URL 更适合作为唯一状态
- 可直接判断是否到达第一张/最后一张原图，无需反查 URL
- 右侧原图面板可以基于同一个索引高亮当前选中的原图，避免两份状态不一致

替代方案是继续保存 URL，再在每次键盘切换时通过 `findIndex()` 反查索引。该方案实现上可行，但状态来源分散，后续更容易出现 URL 重复或列表更新后的同步问题，因此不采用。

### 2. 原图覆盖层独占自己的键盘导航

当原图覆盖层打开时，`ArrowUp` / `ArrowDown` 只用于切换原图，`Esc` 只关闭原图覆盖层，不继续冒泡到审核弹窗。主审核图的 `ArrowLeft` / `ArrowRight` 快捷键只在原图覆盖层关闭时生效。

这样做的原因：
- 避免管理员在放大查看原图时误切换到其他 AI 生成图
- 保持方向键语义清晰：左右控制生成图， 上下控制原图
- 与用户需求中的“放大后看其他原图”直接对齐

替代方案是让原图覆盖层复用 `ArrowLeft` / `ArrowRight`。该方案会与主弹窗现有快捷键冲突，需要额外判断层级，学习成本也更高，因此不采用。

### 3. 原图切换时重置原图缩放状态

每次切换原图预览索引时，调用 `sourceZoom.reset()`，让新原图从适应窗口的初始视图开始展示。

这样做的原因：
- 不同原图尺寸和构图差异很大，沿用上一张图的缩放和平移通常会把新图带到不可见区域
- 重新定位到初始态比“继承上张视图”更可预期

替代方案是保留同一套缩放值与平移偏移。该方案只有在多张原图尺寸完全一致时才有收益，实际场景中误差更大，因此不采用。

### 4. 审核纠错采用有限可逆矩阵，而不是任意状态互转

弹窗操作按钮改为按当前状态显示：
- `pending`: 保持现有 “通过 / 驳回 / 重新生成”
- `approved`: 显示 “改为驳回”
- `rejected`: 显示 “改为通过”
- `regenerate`: 保持只读状态，不在本次变更里开放新的纠错路径

这样做的原因：
- 只覆盖用户明确提出的误判纠错场景
- 不扩大到 `regenerate` 的回退语义，避免引入新的业务讨论
- 能保留当前审核模型的简单性，同时满足最迫切的人工复核需求

替代方案是允许所有非 pending 状态之间任意互转。该方案虽然灵活，但需要重新定义 `regenerate` 的撤销语义和对应提示，超出本次优化范围，因此不采用。

### 5. 首次审核与纠错使用不同的后续导航策略

对 `pending` 图片完成审核后，维持当前“自动跳到下一张 pending”的行为；对已经审核过的图片进行纠错后，弹窗保持在当前图片，不显示“所有图片审核完成”覆盖层。

这样做的原因：
- 首次审核仍然以吞吐效率为主
- 纠错时用户通常在核对当前图是否误判，自动跳走会打断确认过程
- 如果在所有 pending 图片已审核完成后再做纠错，继续触发“全部完成并关闭”会显得突兀

替代方案是所有审核动作都继续跳到下一张 pending。该方案会让已审核图片的纠错体验变差，也会在无 pending 的情况下带来尴尬的关闭行为，因此不采用。

## Risks / Trade-offs

- **快捷键层级更复杂** → 通过“原图覆盖层优先、输入框次之、主弹窗最后”的顺序集中处理，避免同一个按键被多个层同时消费
- **纠错入口增加后更容易误操作** → 通过差异化按钮文案（如“改为驳回”“改为通过”）降低误触概率
- **前端局部状态增多** → 通过索引驱动单一真相源，避免同时维护 URL、索引和选中项三份状态
- **审核完成逻辑更分叉** → 将“pending 首审”和“已审核纠错”明确分支，避免在一个函数里混用相互冲突的自动跳转规则

## Migration Plan

1. 更新审核弹窗状态管理与键盘事件处理
2. 调整操作按钮渲染规则，使 `approved` / `rejected` 支持纠错动作
3. 验证 `PATCH /api/admin/images/[id]/review` 在纠错路径上仍能正确清理或写入 `rejectionReason`
4. 进行桌面端手工回归：原图放大后上下切换、pending 首审自动翻页、approved/rejected 纠错后保持当前图

该变更不涉及数据库迁移。回滚时只需恢复前端交互逻辑与图标按钮渲染。
