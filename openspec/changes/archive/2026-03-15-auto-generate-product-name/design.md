## Context

当前客户新建产品流程里，`products.name` 由用户手动输入，而 `productNumber` 却是服务端自动生成的。与此同时，系统已经有 `batchNumber` 字段，也已经在新建页引入了固定场景选择，但当前创建接口仍把所选场景的 `id` 直接写进 `stylePreference`，这会让产品展示名称和后续 AI prompt 都缺少稳定的人类可读语义。

这个变更横跨客户新建页、创建 API、产品展示以及下游生成链路使用的产品上下文，因此需要先明确“产品身份”应由哪些字段组成，以及这些字段在前后端分别承担什么职责。

## Goals / Non-Goals

**Goals:**

- 新建产品时不再要求客户手动输入产品名称
- 产品创建时由系统生成稳定、可读的 `products.name`
- 名称生成必须基于 `productNumber`、`batchNumber` 和所选场景名称
- 创建链路同时保留“场景 ID”与“场景可读名称”，避免下游只拿到 ID
- 客户列表、详情和后续生成链路读取到的一致都是系统生成后的产品身份信息

**Non-Goals:**

- 本次不改动历史产品的旧名称
- 本次不引入新的编号规则表或复杂命名配置中心
- 本次不重构现有产品编号 `productNumber` 的生成逻辑
- 本次不修改后台审核流程，只保证其展示到新的自动名称

## Decisions

### 1. 产品名称只在服务端生成

创建 API 将忽略客户端传入的 `name`，统一在服务端生成 `products.name`。

生成格式采用：

```txt
{productNumber}-第{batchNumber}批-{sceneName}
```

例如：

```txt
260123-第2批-法式客厅
```

原因：

- `productNumber` 只在服务端生成，名称必须在同一侧拼装才不会出现前后不一致
- 服务端可以统一做场景名称回填、长度控制和兜底策略

备选方案：

- 客户端本地预生成名称：被否决，因为拿不到最终 `productNumber`
- 仅存 `productNumber`，列表现场拼装显示：被否决，因为会让名称缺少稳定真源

### 2. 批次号作为创建参数显式提交，默认值为 1

新建产品表单移除名称输入后，新增或保留批次号输入，并默认值为 `1`。创建 API 接收 `batchNumber`，未传时按 `1` 兜底。

原因：

- 用户明确要求名称中包含批次
- 当前系统虽已有 `products.batchNumber`，但新建流程没有稳定来源

备选方案：

- 从历史最新批次自动推断：被否决，因为推断规则不透明，用户难以预期
- 永远固定为第 1 批：被否决，因为会丢失业务语义

### 3. 固定场景注册表拆分为“场景 ID”和“可读场景语义”

创建链路改为提交 `selectedStyleId` 作为所选场景 ID。服务端根据 `src/lib/scenes.ts` 里的固定场景注册表解析名称，并做两件事：

- `products.selectedStyleId` 保存场景 ID
- `products.stylePreference` 保存场景名称或其可读语义，供列表展示和 worker prompt 使用

原因：

- 下游 `stylePreference` 当前被当作自然语言语义使用，不能继续写入场景 ID
- 自动名称需要使用可读场景名，而不是数据库主键

备选方案：

- 只保存场景 ID，运行时再到处查名称：被否决，因为会增加所有消费方复杂度
- 只保存场景名称，不保存场景 ID：被否决，因为后续无法准确追踪用户选择的是哪个场景

### 4. 创建接口在过渡期兼容旧客户端

为了避免前后端发布不同步导致创建失败，API 在短期内仍允许接收旧的 `stylePreference` 字段，但优先使用新的 `selectedStyleId`。如果只能拿到旧字段且它匹配模板 ID，则按模板解析；如果是普通文本，则仅作为兼容回退。

原因：

- 当前仓库和运行环境存在并行改动，接口需要有一定兼容性

## Risks / Trade-offs

- [场景模板 ID 与可读名称混用] → 在 API 层统一解析并落库，避免 worker 直接消费 ID
- [批次默认值与真实业务批次不一致] → 默认给 `1`，同时在 UI 暴露可编辑批次字段
- [名称格式过长] → 服务端生成时限制场景名称片段长度，必要时截断
- [旧客户端仍传 `name`] → API 忽略客户端名称，保证系统真源唯一

## Migration Plan

1. 更新客户新建产品 UI，移除名称输入并补充批次输入
2. 更新创建 API，支持 `selectedStyleId`、`batchNumber`，并服务端自动生成名称
3. 更新列表/详情读取结果，直接展示已生成的产品名称
4. 补充创建链路测试与产品展示回归测试

回滚策略：

- 若新流程出现问题，可回滚前端恢复名称输入
- API 端保留对旧 payload 的兼容，便于前后端分步回退
